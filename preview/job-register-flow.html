🔄 전체 흐름

🔁 자기 발신 메시지 무한 루프 방지

업무 등록 → LINE 발송 → LINE Webhook → 큐 적재 → Claude 파싱 → 등록 대기 (← 자기 메시지를 다시 등록 대기에 띄우면 안 됨!)
시그니처 생성 (발송 시)
sha256(roomId + ":" + msgText) 의 앞 16자.
KV 키: line_outbound_sig:<hash>
값: {jobId, sentAt, by}
TTL: 10분 (그 안에 webhook 으로 돌아옴)
Webhook 수신 시 (line-webhook.js)
메시지 수신 → 동일 해시 계산 → KV 조회
- hit: 큐에 적재하되 processedStatus='self_echo'
- 미스: 정상 적재
파싱 cron 은 self_echo 상태인 메시지 모두 skip.
왜 ID 기반이 아니라 해시 기반?
LINE Messaging API 의 push 응답은 messageId 를 즉시 주지 않고, webhook 의 source.userId 는 봇 자신이지만 sender 정보가 messageId 와 결합되지 않음. 본문 해시가 양방향 식별에 가장 안정적.
대시보드에서 자기 메시지 보기
등록 대기에는 안 보임. 대시보드 → 작업 상세 → 발송 이력 탭에 시각 + 메시지 본문 + 시그니처 노출. 작업 ↔ LINE 메시지 양방향 추적.

📌 설계 메모 — 핵심 결정 사항

A. 매장 검색 결과의 진행중 업무 노출
매장 선택 즉시 (스텝 2 안에서) 같은 매장의 미완료 작업을 LINE 채팅 기록처럼 시간순으로 보여줌. 사용자가 "기존 업무 갱신" vs "신규 등록" 명시적 선택.
B. 카테고리별 폼 분기
스텝 3 의 필드가 카테고리에 따라 동적. 예: AS = 증상/현장상황/장비 / 오픈 = 일정/투입장비 / 상담 = 자유 텍스트. 기존 editNewopen 폼 컴포넌트 재사용.
C. LINE 그룹 자동 매칭
line_config.roomMap 의 카테고리 ↔ 룸 매핑을 역참조: pos_as→A/S, van_doc→밴업무 공유, schedule→일정. 사용자가 override 가능.
D. LINE 발신 후 처리
발신 메시지를 job.outboundMessages[] 에 기록. 추후 같은 그룹에 수신된 답장 메시지는 job.lineReplies[] 로 자동 연결 (replyToJobId 추론).
E. 진행 단계 표준화
카테고리별 단계 흐름 정의: AS = 접수→발주→방문→완료 / 오픈 = 계약→공사→설치→가오픈→오픈 / 밴서류 = 신청→심사→완료. 기존 AS_STATUS_FLOW 패턴 확장.
F. 등록 진입점
대시보드 상단 [+ 새 업무 등록] 큰 버튼 / 매장 상세 화면 [+ 이 매장 업무 등록] / 일정 캘린더 빈 셀 클릭. 3 곳 모두 같은 wizard 로 진입.
G. LINE 메시지 표준 포맷
[카테고리] 매장명 / 👤 담당 / 📅 처리요청일 / ━━ / 요청내용 의 5요소 고정. 사용자가 메시지 박스를 직접 편집하면 그 override 가 우선. 폼 필드 변경시 사용자 편집을 안전 보존(autoBuild 일시 정지).
H. 거래처 담당 = 매장 contacts 와 연동
매장 데이터의 store.contacts[] (이름·직책·전화·primary 플래그) 와 자동 양방향 동기화: 등록시 입력값을 매장 DB 에도 추가. 매장에 이미 contacts 있으면 dropdown 으로 자동 채움 + "신규 추가" 옵션. 같은 통화 상대를 매번 다시 입력하지 않게.
I. 자기 발신 메시지 루프 차단
발송 직전 본문 해시(sha256 앞 16자)를 KV line_outbound_sig:<hash> 에 TTL 10분으로 저장. line-webhook.js 가 같은 해시 매칭 시 큐에 적재하되 processedStatus='self_echo' 마크 → line-parse-cron.js 가 skip. 등록 대기 화면에 우리 발송 메시지가 다시 뜨는 사고 방지.
J. 외부 → 내부 동일 매장 메시지 자동 연결
파싱 결과의 store 가 우리가 최근 발송한 jobId 의 매장과 일치하면 등록 대기 항목에 "기존 작업과 연결?" 추천 표시. 답장 ↔ 작업 자동 trace 강화 (J 는 다음 단계, 일단 I 만 우선 구현).
