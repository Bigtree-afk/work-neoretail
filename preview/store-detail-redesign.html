📐 탭 구조 설계 원칙

A. 첫 탭 = 진행 업무 ⚠
매장을 보는 가장 큰 이유는 "지금 뭐 해야 하지?". 진행 업무를 0번째 탭으로. 카운트 chip 빨간 펄스 = 즉시 시선 끌기.
B. 매장 정보는 2번 탭
상호·연락·POS 환경 등 기본 데이터. 빠른 확인용. 정보 변경 버튼은 헤더에 분리.
C. 카테고리별 이력 탭
오픈/AS/밴서류/출고/라벨/상담 — 각 카테고리마다 별도 탭. 카운트 chip 으로 누적 건수 한눈에. 필터(전체/완료만/최근 등)로 세분화.
D. 출고/택배는 묶기
장비출고 + 택배 + 라벨지 발송은 "📦 출고/택배" 한 탭으로. 운영 관점에서 같은 흐름. 안에서 다시 필터.
E. 설치 장비 별도 탭
store.equipment[] (Plan B DB) 인스턴스 단위 관리. 추가/편집/상태변경/이전. 작업 기록과 분리된 자산 관점.
F. 메모/이력 마지막 탭
매장 자체 메모 + 정보 변경 이력 + 작업 메모 통합. 모두의 자유 메모장.
⚙ 구현 가이드 — 탭은 모두 같은 모달 안에서 lazy 렌더링 (active 탭만 DOM 생성). 데이터는 getJobs().filter(j => sameStore(j) && j.category === ...) 기반. URL hash 로 탭 상태 보존 가능 (#tab=as).
